01Goal & Problem Statement▼

The Problem

Export: CX books vessels without reliably accounting for barge/rail transit times, terminal YOT, and CCO. This causes YOT refusals and CCO misses leading to rolled cargo.

Import: CX needs to plan inland transport from discharge ports (Rotterdam/Antwerp) to German inland depots, coordinating barge/rail departures with vessel ETD dates and customs deadlines.

The Goal

A single unified tool covering both directions: Import (port → inland depot → customer delivery) and Export (customer loading → inland depot → port terminal → vessel). CX enters shipment details and immediately sees feasible options with all deadlines calculated.

05Calculation Logic▼

Export Logic

Loading date → find next barge ETD → EAT (ETD + transit) → CCO boundary (EAT + buffer) → Latest vessel ETA (EAT + YOT − 1 inclusive).

Export Example

Barge departs Mon 02/03 → arrives Thu 05/03 (EAT). YOT=5: Day 1=Thu…Day 5=Mon 09/03. → Book vessel where CCO ≥ Thu 05/03 and ETA ≤ Mon 09/03.

Import Logic

Vessel ETD → find next terminal departures (barge/rail) → depot arrival (ETA) → customer delivery (next business day) → order deadline (5–7 biz days before ETD) → customs deadline (Rotterdam only).

Import Example

Vessel ETD Wed → next barge from Rotterdam on Thu → arrives at depot Mon → customer delivery Tue. Order deadline = previous week. For future bookings, 7-day lead time; for discharged cargo, 5-day lead time.

06Feasibility Rules▼

Export — Hard Constraints

HC-1
Vessel CCO ≥ Planning ETA
HC-2
Vessel ETD ≥ Planning ETA + YOT days
HC-3
Barge ETD ≥ customer loading date

Import — Planning Rules

IR-1
Transport order must be placed before order deadline
IR-2
Rotterdam customs docs must be submitted before customs deadline
IR-3
If order deadline passed → ISR required to check availability
07Output Design▼

Export Output

Customs deadline banner (loading + 3h) → depot options (ranked pills) → max 2 schedule cards per depot showing vessel booking window (earliest CCO, latest vessel ETA).

Import Output

Summary bar → terminal info → schedule cards split into "Plannable" (book directly) and "ISR Required" (deadline passed) sections with copy-to-clipboard summaries.

09Assumptions▼
  • Export uses last day of arrival range (conservative) for transit calculations.
  • Import uses single ETD/ETA day per schedule from schedule data.
  • All export calculations use calendar days; import uses business day logic for order deadlines.
  • German public holidays (2025–2027) are accounted for in import business-day calculations and flagged in export transit windows.
  • Tool is advisory only — does not create bookings.
